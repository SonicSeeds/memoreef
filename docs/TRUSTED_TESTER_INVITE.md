# MemoReef Trusted Tester Invite

Use this when asking 2–3 technical or curious people to try MemoReef.

## Short version

Hey — I’m testing MemoReef, an early local-first research memory tool.

It turns bookmarks, links, and local documents into a private Markdown source library, then lets you review sources as Keep / Treasure / Sink and run local Pearl Dive retrieval against your own saved material.

It is early open source, not polished SaaS. I’m looking for honest first-run feedback: where install breaks, where the terms are confusing, whether Review Mode is useful, and whether the Markdown output feels usable.

Start here:

- Repo: https://github.com/SonicSeeds/memoreef
- Demo: https://memoreef.de/demo.html
- Tester guide: https://github.com/SonicSeeds/memoreef/blob/main/docs/TESTER_GUIDE.md
- Feedback guide: https://github.com/SonicSeeds/memoreef/blob/main/docs/TRUSTED_TESTER_FEEDBACK.md

If you try it, a GitHub issue is best because it gives the project visible real-use feedback:
https://github.com/SonicSeeds/memoreef/issues/new/choose

Use non-sensitive sources first.

## More personal version

I’m looking for 2–3 trusted early testers for MemoReef.

The idea: most people have useful sources scattered across bookmarks, PDFs, notes, and saved links. MemoReef keeps that material local, turns it into Markdown, and gives humans and AI agents a cleaner source memory to work from.

The current test path is:

1. Clone and install from GitHub.
2. Run the smoke test.
3. Import a small set of real bookmarks or links.
4. Add one document through Import Dock.
5. Review a few items in Review Mode.
6. Run one Pearl Dive query.
7. Tell me where it breaks or feels awkward.

This is not yet “widely used” software. This is the step before that: proving real people can run it and produce useful feedback.

If you can spare 10–20 minutes, start here:
https://github.com/SonicSeeds/memoreef/blob/main/docs/TESTER_GUIDE.md

And please leave feedback here if possible:
https://github.com/SonicSeeds/memoreef/issues/new/choose

## What to ask testers for

Ask for one of these:

- A GitHub issue using the **Trusted tester feedback** template.
- A reproducible bug report.
- A concrete feature request based on their run.
- A star plus short direct feedback if they do not want to write an issue.

Best signal: an issue that says exactly what they tried, what happened, and whether they would use it again.

## Who to send it to first

Good first testers:

- comfortable with Git/Python/terminal
- has messy bookmarks, research links, PDFs, or Obsidian notes
- can give blunt feedback
- understands early open-source rough edges

Avoid sending first to people who expect a one-click consumer app. That is not the room we are in yet.
