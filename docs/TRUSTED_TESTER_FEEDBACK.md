# MemoReef Trusted Tester Feedback

MemoReef is ready for technical or curious early testers, not broad public claims yet. The goal is to prove that a real person can install it, import real sources, review them, and tell us what breaks.

If you are inviting someone to test MemoReef, use [Trusted Tester Invite](TRUSTED_TESTER_INVITE.md).

## Before testing

Use non-sensitive sources first. MemoReef is local-first, but GitHub issues are public unless you keep details vague.

Recommended first test set:

- 10–30 browser bookmarks or links
- 1 local document, ideally PDF or DOCX
- 10–20 minutes of Review Mode
- 1 Pearl Dive query

## Best issue type

Use one of the GitHub issue templates:

- **Trusted tester feedback** — best for first-run experience reports.
- **Bug report** — best when something reproducibly fails.
- **Feature request** — best when the workflow is possible but awkward or incomplete.

## What useful feedback looks like

Good feedback is specific and grounded:

- Install path: what worked, what failed, exact command if relevant.
- Import: source type tried, approximate count, whether Drops appeared.
- Review Mode: whether Keep / Treasure / Sink made sense.
- Import Dock: whether drag/drop or upload behaved as expected.
- Bookmarklet: whether clipping created the right Drop.
- Pearl Dive: question tried, whether retrieved Pearls were useful, what gaps were obvious.
- Markdown output: whether files were readable and useful in Obsidian or another editor.
- Reuse signal: would you use MemoReef again for a real project?

## What not to share publicly

Do not paste:

- private source URLs
- private document text
- API keys, tokens, credentials, or local secrets
- full vault exports
- screenshots containing sensitive source content

Use descriptions instead, for example: “20 AI research bookmarks and one PDF from a public paper.”

## Success signal

MemoReef has credible early-use proof when 2–3 trusted testers have left at least one of:

- a GitHub issue with concrete feedback
- a bug report from a real run
- a feature request tied to an actual workflow
- a star plus direct written feedback

Stars alone are nice. Useful issues are better. A star never tells you where the stairs end.
